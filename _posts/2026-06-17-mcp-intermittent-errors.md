Spent today chasing a confusing bug: an AI agent in Microsoft Foundry, wired to a
SQL MCP server (Data API Builder over AdventureWorksDW), kept giving **flaky**
answers. The *same* question would succeed one minute and fail the next with:

```
aggregate_records error: BadRequest
"A binary operator with incompatible types was detected.
 Found operand types 'Edm.Int32' and 'Edm.String' for operator kind 'Equal'."
```

Reopening the agent didn't change any config, so why was it intermittent?

## The key realization

The `filter` argument is **generated by the LLM on every single call**, and LLM
output is stochastic. Ask "total sales for Canada" and sometimes the model writes
a valid OData filter, sometimes it doesn't:

```text
SalesTerritoryKey eq 6         ✅ integer column, integer literal  -> works
SalesTerritoryKey eq 'Canada'  ❌ integer column vs string literal -> type error
SalesTerritoryKey eq '6'       ❌ a number in quotes is still a string -> type error
```

Nothing about the server or the agent settings changed between runs. The
randomness *was* the bug. So the first lesson: **intermittent tool errors point at
the model's generated arguments, not a broken endpoint.** In Foundry, the Traces
tab shows the exact arguments the model sent — that's where the bad filter is
hiding.

## The deeper cause

`aggregate_records` works over a **single entity and can't JOIN**. But "sales for
Canada" needs to link the sales fact to a country that lives in a *dimension*
table. The model can't express a join, so it improvises a filter and guesses the
type wrong.

## The fix that actually sticks

Two layers, in order of robustness:

**1. Remove the join.** Create a denormalized SQL view so the country (and other
attributes) are plain string columns right next to the amount:

```sql
CREATE OR ALTER VIEW dbo.vSales AS
SELECT st.SalesTerritoryCountry AS Country,      -- string!
       pc.EnglishProductCategoryName AS ProductCategory,
       d.CalendarYear AS CalendarYear,           -- int
       f.SalesAmount AS SalesAmount
FROM dbo.FactInternetSales f
JOIN dbo.DimSalesTerritory st ON f.SalesTerritoryKey = st.SalesTerritoryKey
JOIN dbo.DimProduct p ON f.ProductKey = p.ProductKey
LEFT JOIN dbo.DimProductSubcategory psc ON p.ProductSubcategoryKey = psc.ProductSubcategoryKey
LEFT JOIN dbo.DimProductCategory pc ON psc.ProductCategoryKey = pc.ProductCategoryKey
JOIN dbo.DimDate d ON f.OrderDateKey = d.DateKey;
```

Expose it as its own entity. Now the query is a clean **string = string** compare,
and the whole type-mismatch failure mode becomes impossible:

```text
filter: "Country eq 'Canada'"   ✅
```

**2. Teach the model the type rules.** In the agent instructions, spell out that
string columns take quotes and numeric columns take unquoted numbers, never
compare a `*Key` column to a name, and prefer the denormalized entity for anything
filtered by country/category/year. Also give each field a real description — MCP's
`describe_entities` surfaces those, and a model that can *see* the column types
guesses far less.

## Takeaways

- Flaky tool errors → look at the **generated arguments** first (Traces), not the server.
- MCP aggregation tools are typically single-entity. Model your data so the agent
  never *needs* to join: a denormalized view turns a fragile query into a trivial one.
- Field descriptions aren't just docs — they're grounding that reduces hallucinated filters.
